<objective>
Build the three secondary pages — confirmation, unsubscribe, and privacy — and finalize the application route map. Purpose: These pages close the email-triggered flows (double opt-in confirmation and RFC 8058 unsubscribe). Without them, every backend email leads to a dead link.
Output: Three page components, their specs, and an updated `app.routes.ts` that resolves all four lazy routes.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| URL query param → component @Input | `?token=X` from email link enters the Angular app via router input binding |
| Component → API | Token value passed as GET query parameter to backend |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-03-01 | Information Disclosure | ConfirmComponent / UnsubscribeComponent error states | mitigate | Error messages are fixed Dutch strings; the `HttpErrorResponse` status, body, or message is never rendered in the template or logged in production |
| T-03-03-02 | Elevation of Privilege | Token in browser URL (`?token=X`) | accept | Tokens are single-use and short-lived (24 h TTL per Phase 2 design). The browser history exposure of a single-use token is an accepted residual risk — the attacker would need access to browser history AND the token would need to be unused |
| T-03-03-03 | Spoofing / XSS | `privacy.component.html` static content | accept | All content is static Dutch copy authored by the developer; no user input or API response is rendered; no `[innerHTML]` used |
| T-03-03-04 | Tampering | Open redirect via `routerLink` | accept | All `routerLink` values (`/`, `/privacy`) are hardcoded string literals; no user-controlled values flow into router navigation |
</threat_model>
