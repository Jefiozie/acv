<objective>
Implement the full `StatefulStack`: a DynamoDB `TableV2` with single-table design (GSI1 for confirmed-subscriber fan-out, GSI2 for unsubscribe token lookup), an SES `EmailIdentity` for `acv-aanhanger.nl`, and an SES `ConfigurationSet` with suppression enabled. Deploy to AWS and verify the resources exist.

Purpose: The DynamoDB table and SES identity are the two stateful resources that every other stack depends on. They must exist before Phase 2 Lambda code can write subscribers or send emails. Both carry CRITICAL data-safety constraints: the table must survive accidental stack operations (`RemovalPolicy.RETAIN` + stack termination protection), and the SES identity must be verified via DNS before emails can be sent (DNS step follows in plan 01-03).

Output: `AcvStateful` CloudFormation stack deployed in AWS; DynamoDB table with two GSIs visible in Console; SES identity at "Pending" state awaiting DNS records.
</objective>

    PUBLIC API (must match what bin/app.ts already expects):
    - `public readonly table: dynamodb.TableV2`
    - `public readonly configSetName: string`

    DYNAMODB TABLE — instantiate `new dynamodb.TableV2(this, 'SubscriptionsTable', { ... })`:
    - `tableName: undefined` — let CDK generate the name (NEVER hardcode)
    - `partitionKey: { name: 'PK', type: dynamodb.AttributeType.STRING }`
    - `sortKey: { name: 'SK', type: dynamodb.AttributeType.STRING }`
    - `billing: dynamodb.Billing.onDemand()` — no capacity planning, scales to zero
    - `pointInTimeRecovery: true` — ASSUMED prop name for CDK 2.257.0 TableV2. If `tsc --noEmit`
      rejects this prop, switch to `pointInTimeRecoverySpecification: { pointInTimeRecoveryEnabled: true }`
      (the STACK.md alternative). The tsc verification step in this task MUST catch this.
    - `removalPolicy: cdk.RemovalPolicy.RETAIN` — CRITICAL: never destroy subscriber data on
      stack updates or `cdk destroy`. This is one half of the two-safeguard pattern.
    Assign the result to `this.table`.

    GSI1 — "All confirmed subscribers for a township" (sparse GSI, only populated on confirm):
    Call `this.table.addGlobalSecondaryIndex({ ... })`:
    - `indexName: 'GSI1'`
    - `partitionKey: { name: 'GSI1PK', type: dynamodb.AttributeType.STRING }` — value: `TOWNSHIP#<townshipId>`
    - `sortKey: { name: 'GSI1SK', type: dynamodb.AttributeType.STRING }` — value: `CONFIRMED#<email>`
    - `projectionType: dynamodb.ProjectionType.INCLUDE`
    - `nonKeyAttributes: ['confirmed', 'frequency']` — Phase 2 checker Lambda needs both to filter by frequency

    GSI2 — "Subscriber by unsubscribe token" (always populated at subscribe time):
    Call `this.table.addGlobalSecondaryIndex({ ... })`:
    - `indexName: 'GSI2'`
    - `partitionKey: { name: 'GSI2PK', type: dynamodb.AttributeType.STRING }` — value: `TOKEN#<unsubscribeToken>`
    - `projectionType: dynamodb.ProjectionType.KEYS_ONLY` — Phase 2 only needs PK+SK to DeleteItem

    SES DOMAIN IDENTITY — instantiate `new ses.EmailIdentity(this, 'SenderIdentity', { ... })`:
    - `identity: ses.Identity.domain('acv-aanhanger.nl')` — domain identity (not email identity)
      covers ALL addresses on the domain and enables Easy DKIM automatically.
    - Do NOT specify `mailFromDomain` — use SES default for Phase 1.
    - DKIM CNAME records are NOT exposed as CDK outputs; they are retrieved via AWS CLI in plan 01-03.

    SES CONFIGURATION SET — instantiate `new ses.ConfigurationSet(this, 'SesConfigSet', { ... })`:
    - `configurationSetName: 'AcvNotifications'` — fixed name so Lambda env var is stable
    - `suppressionReasons: ses.SuppressionReasons.BOUNCES_AND_COMPLAINTS`
    - `reputationMetrics: true` — enables bounce/complaint rate metrics in SES console
    - `sendingEnabled: true`
    Assign `configSet.configurationSetName!` to `this.configSetName`. The `!` is safe here
    because we just set a literal `configurationSetName` string above.

    CDK OUTPUTS — add three CfnOutput constructs:
    - id `TableName`, value `this.table.tableName` — consumed by BackendStack Lambda env vars
    - id `TableArn`, value `this.table.tableArn` — for IAM policy reference
    - id `ConfigSetName`, value `this.configSetName` — consumed by BackendStack Lambda env vars

    After writing the file, run `cd infrastructure && npx tsc --noEmit` to validate.
    If it fails on `pointInTimeRecovery`, switch to the `pointInTimeRecoverySpecification`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CDK (local) → AWS CloudFormation | CDK deploy sends templates + IAM changes; deploy role must be scoped |
| DynamoDB table → RemovalPolicy | Accidental DESTROY setting would silently queue data deletion on next stack update |
| SES identity → DNS | Domain identity unverified until DNS records added; sending fails if not verified |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-02a | Tampering | DynamoDB RemovalPolicy | mitigate | Two-safeguard pattern: `removalPolicy: cdk.RemovalPolicy.RETAIN` on TableV2 + `terminationProtection: true` on AcvStateful stack in bin/app.ts. Task 2 Step E explicitly verifies termination protection is `true` after deploy. |
| T-01-02b | Tampering | DynamoDB RemovalPolicy | mitigate | `cdk deploy AcvStateful` with `--require-approval never` is safe ONLY because RETAIN means no destructive resource changes; for security group/IAM changes in future, require approval |
| T-01-03a | Information Disclosure | SES identity credentials | mitigate | No SMTP passwords or DKIM private keys are stored in CDK code; SES manages DKIM keys; `noreply@acv-aanhanger.nl` address is not a secret |
| T-01-04a | Elevation of Privilege | CDK deploy IAM role | mitigate | Use `CDK_DEFAULT_ACCOUNT` env var and a scoped IAM role; never use root credentials; verify with `aws sts get-caller-identity` before deploying |
| T-01-SC | Tampering | npm installs | mitigate | Packages already verified in plan 01-01 Task 1; no new packages in this plan |
</threat_model>
