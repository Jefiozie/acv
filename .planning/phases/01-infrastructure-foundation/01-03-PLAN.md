<objective>
Start the external approval timers. This plan has two goals that MUST be completed before Phase 2 can send emails:

1. **DNS records**: Extract the three CDK-generated DKIM CNAME tokens from SES, then add all required DNS records (DKIM × 3, SPF × 1, DMARC × 1) at the domain registrar. DNS propagation takes up to 72 hours.

2. **SES production access**: File an AWS Support case requesting SES production access (removal from sandbox). This has a 24–72 hour approval SLA and blocks Phase 2 from sending to unverified addresses.

Both timers start when this plan completes. Neither can be skipped or deferred — they are Phase 2's critical path.

Purpose: SES sandbox mode silently rejects emails to non-verified addresses. DKIM/SPF/DMARC records must verify before SES counts the domain as authenticated. Both are external approval processes outside CDK control.

Output: DNS records submitted to registrar; SES support ticket filed with ticket number recorded.
</objective>

<task type="auto">
  <name>Task 1: Extract DKIM tokens and format DNS records</name>
  <files>
    .planning/phases/01-infrastructure-foundation/dns-records.txt
  </files>
  <action>
    STEP A — Retrieve DKIM tokens from SES:
    Run:
      `aws sesv2 get-email-identity --email-identity acv-aanhanger.nl --query 'DkimAttributes.Tokens' --output json`

    This returns a JSON array of three token strings (UUIDs), e.g.:
      `["abc123xyz", "def456uvw", "ghi789rst"]`

    DKIM CNAME records are NOT in CDK outputs — they come from the SES API after deploy.
    If this command returns an empty array, the EmailIdentity was not successfully deployed
    in plan 01-02. Re-run `cdk deploy AcvStateful` before proceeding.

    STEP B — Also retrieve the current identity verification status:
    Run:
      `aws sesv2 get-email-identity --email-identity acv-aanhanger.nl --query '{VerificationStatus:VerificationStatus,DkimStatus:DkimAttributes.Status,DkimEnabled:DkimAttributes.SigningEnabled}' --output json`

    Expected at this point: VerificationStatus=PENDING, DkimStatus=NOT_STARTED.
    This is correct — DNS records have not been added yet.

    STEP C — Format all DNS records to add:
    Write the following to `.planning/phases/01-infrastructure-foundation/dns-records.txt`
    (substitute actual token values from Step A for <token1>, <token2>, <token3>):

    --- DKIM CNAME records (×3) ---
    Host:  <token1>._domainkey.acv-aanhanger.nl
    Type:  CNAME
    Value: <token1>.dkim.amazonses.com

    Host:  <token2>._domainkey.acv-aanhanger.nl
    Type:  CNAME
    Value: <token2>.dkim.amazonses.com

    Host:  <token3>._domainkey.acv-aanhanger.nl
    Type:  CNAME
    Value: <token3>.dkim.amazonses.com

    --- SPF TXT record (×1) ---
    Host:  acv-aanhanger.nl  (root / @)
    Type:  TXT
    Value: "v=spf1 include:amazonses.com ~all"
    Note:  If a TXT record already exists on this host, APPEND to it rather than
           replacing. Multiple TXT values are allowed by DNS spec. Example:
           "v=spf1 include:amazonses.com include:someotherprovider.com ~all"

    --- DMARC TXT record (×1) ---
    Host:  _dmarc.acv-aanhanger.nl
    Type:  TXT
    Value: "v=DMARC1; p=none; rua=mailto:postmaster@acv-aanhanger.nl; sp=none; adkim=r; aspf=r"
    Note:  p=none is monitoring mode — safe for initial setup. After Phase 4 confirms all
           email passes DKIM/SPF alignment, upgrade to p=quarantine or p=reject.

    Print all records to the console prominently so the next task (human-action) can
    copy-paste them directly into the domain registrar.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| DNS registrar interface | Human-only step; DKIM values travel from SES API output to registrar UI |
| AWS Support case | Human-only step; use case description must not include subscriber PII |
| dns-records.txt (local file) | Contains DKIM token values — not secrets but sensitive (spoofed CNAME could enable email spoofing if registrar is compromised) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-05 | Spoofing | DKIM token values | mitigate | Token values come directly from `aws sesv2 get-email-identity` (authenticated AWS API call); dns-records.txt stays in .planning/ (gitignored if contains sensitive data — but DKIM tokens are public DNS values so this is acceptable) |
| T-01-06 | Tampering | SPF TXT record | mitigate | Append to existing SPF record rather than replace, to avoid invalidating other authorized senders; DMARC p=none is monitoring mode (non-destructive) |
| T-01-07 | Repudiation | SES production access request | mitigate | Record ticket number in ses-ticket.txt; include in plan SUMMARY for audit trail |
| T-01-08 | Elevation of Privilege | SES sandbox bypass | accept | SES sandbox is a safety net — approved production access is legitimate for a double opt-in transactional email service; risk is low given suppression list + double opt-in |
| T-01-SC | Tampering | npm installs | accept | No new package installs in this plan |
</threat_model>
